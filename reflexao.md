# Reflexão — Roteiro 03

## 1. Stubs e Skeletons

O **stub** (lado cliente) e o **skeleton** (lado servidor) são os componentes que tornam possível a transparência do RPC. Na Tarefa 2, o stub (`_stub_chamar`) realizou quatro etapas de forma transparente ao código chamador: serialização da chamada em JSON (*marshalling*), adição do cabeçalho de tamanho (*framing*), envio via socket TCP e deserialização da resposta (*unmarshalling*). O skeleton, por sua vez, fez o caminho inverso: leu os bytes recebidos, decodificou a mensagem, localizou a função correta na *dispatch table* e devolveu o resultado serializado. Sem esses componentes, o programador precisaria escrever toda essa lógica de rede e serialização manualmente em cada chamada, tornando o código de negócio inseparável do código de infraestrutura de comunicação — exatamente o problema que o RPC veio resolver.

## 2. REST não é RPC

A diferença fundamental de modelagem está na **unidade de abstração**: o RPC expõe *ações* (funções nomeadas), enquanto o REST expõe *recursos* (entidades identificadas por URI). Na Tarefa 1, o cliente chamou `proxy.calcular("soma", 10, 3)` — uma invocação de procedimento, onde o verbo está embutido no nome da função. Na Tarefa 3, o mesmo conceito foi modelado como `POST /calculos` com payload JSON — o verbo HTTP (`POST`) e o recurso (`/calculos`) estão separados e padronizados. Fielding critica o RPC sobre HTTP justamente porque ele ignora essa interface uniforme: ao criar um endpoint `/calcular` que responde a `POST`, o servidor está usando o HTTP apenas como transporte, não como protocolo semântico. A restrição de interface uniforme do REST exige que os verbos HTTP expressem a operação sobre o recurso, não que o nome da função seja codificado na URI.

## 3. Evolução de Contrato

O Protocol Buffers (Protobuf) lida com evolução de contrato através da compatibilidade de campos por número de tag. Se o campo `resultado` for `double` (tag `1`) e for adicionado `unidade: string` com a tag `3`, clientes antigos simplesmente ignoram o campo desconhecido ao deserializar — o binário permanece compatível. Isso é chamado de *forward compatibility*: o cliente antigo não quebra ao receber uma mensagem nova que contém campos extras. No REST, sem schema formal, a mesma mudança seria feita adicionando o campo `"unidade"` ao JSON retornado. Clientes bem escritos também ignorariam o campo desconhecido, mas isso depende de convenção e disciplina do desenvolvedor — não há garantia automática. Se o campo `resultado` fosse *renomeado* de `double` para `string`, o Protobuf detectaria a incompatibilidade de tipo na compilação; no REST, o cliente só descobriria o problema em runtime ao tentar usar o valor.

## 4. Escolha de Tecnologia

Para a **API pública de pagamentos** (consumida por apps de terceiros), recomendaria **REST com HTTP/1.1 e JSON**. A razão principal é a universalidade: qualquer linguagem, framework ou ferramenta (curl, Postman, navegador) consegue consumir uma API REST sem dependências extras. A ausência de schema obrigatório facilita a evolução incremental sem forçar os parceiros a regenerar stubs. Para a **comunicação interna entre os 10 microsserviços**, recomendaria **gRPC**. A tipagem forte do `.proto` garante contratos verificados em tempo de compilação, eliminando erros de integração descobertos apenas em produção. A serialização binária com Protobuf reduz significativamente o tamanho dos payloads e o overhead de parsing em relação ao JSON, o que é crítico em comunicação de alta frequência entre serviços internos. O suporte nativo a streaming bidirecional do HTTP/2 também é vantagem para padrões de comunicação mais complexos entre microsserviços.

## 5. Conexão com Labs Anteriores

O Lab 04 mostrou que transparência excessiva pode induzir o desenvolvedor a tratar chamadas remotas como se fossem locais, ignorando diferenças fundamentais de latência, falha e confiabilidade. No contexto do RPC, essa transparência é especialmente perigosa: ao chamar `proxy.calcular("soma", 10, 3)` na Tarefa 1, o código parece idêntico a chamar uma função local, mas envolve serialização XML, abertura de conexão TCP, transmissão de rede e deserialização — operações que podem levar centenas de milissegundos e falhar por razões completamente diferentes das falhas locais (timeout de rede, servidor reiniciado, particionamento). Uma decisão de design errada comum é chamar um método RPC dentro de um loop intenso ou de uma transação de banco de dados, sem perceber que cada iteração pode demorar 100ms e falhar de forma não-determinística. A transparência do RPC esconde esse custo; um bom desenvolvedor precisa deliberadamente "quebrar" a ilusão de localidade e tratar chamadas remotas com mecanismos explícitos de retry, timeout e circuit breaker.
